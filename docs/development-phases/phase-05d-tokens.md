## Phase 05d: add token capability with required analysis tokens and option tokens

Build a token replacement system for Guidance Blocks and assembled guidance output.

This task has two parts:

1. Update the analysis output contract so required organization/content tokens are produced by the initial AI analysis call.
2. Add deterministic token replacement during guidance assembly.

Do not rewrite the default Guidance Block text in this task. This task creates the token capability so the default text can be improved afterward.

## No backward compatibility requirement

This project is still in active MVP development.

Do not add compatibility fallbacks for older AI response shapes created during earlier development.

Old run artifacts may be discarded and regenerated.

If an existing run artifact does not include the required token fields, it may be treated as invalid/incomplete for this phase.

Do not add complex fallback logic to support abandoned intermediate formats.

## Token categories

There are two categories of tokens.

### 1. Analysis tokens

Analysis tokens come from the initial scan/analysis AI call.

They describe the organization, scanned content, and generated voice/tone profile.

They must be present in the stored analysis/guidance artifact for a completed run.

Examples:

```text
{{organizationName}}
{{organizationShortName}}
{{detectedOrganizationType}}
{{voiceTonePreviewSummary}}
{{voiceToneFullGuidance}}
```

The coding agent should not invent these values during workbench rendering.

If the value requires interpretation of scanned source material, it belongs in the analysis AI response.

### 2. Option tokens

Option tokens come from deterministic workbench state/config.

They describe the user’s current selections.

They are not generated by AI.

Examples:

```text
{{guidanceTaskLabel}}
{{lengthDetailLabel}}
{{readingLevelLabel}}
{{readingLevelDescription}}
{{bestPracticePackLabel}}
```

These should be derived from the selected option values and the default workbench configuration.

## Required analysis output shape

Revise the initial AI analysis call/output schema so the stored analysis artifact includes:

```json
{
  "organization": {
    "name": "Bill & Melinda Gates Foundation",
    "shortName": "Gates Foundation",
    "detectedType": "foundation",
    "websiteUrl": "https://www.gatesfoundation.org"
  },

    "fullGuidance": "Write with moral urgency grounded in data. The Gates Foundation's voice combines unflinching honesty about problems with conviction that solutions exist and are within reach..."
  }
}
```

Adjust exact nesting to match the existing artifact structure, but preserve the required fields and meanings.

## Required analysis fields

### `organization.name`

Full organization name detected from the scan or canonical source.

Example:

```text
Bill & Melinda Gates Foundation
```

This powers:

```text
{{organizationName}}
```

### `organization.shortName`

A shorter natural name for repeated use in assembled guidance.

Example:

```text
Gates Foundation
```

This powers:

```text
{{organizationShortName}}
```

This must be generated by the initial analysis AI call. The workbench should not try to invent it with string manipulation.

If no natural short name exists, the analysis AI call should return the full name as `shortName`.

### `organization.detectedType`

A simple detected organization type.

Example values may include:

```text
foundation
nonprofit
school
association
public_agency
unknown
```

This powers:

```text
{{detectedOrganizationType}}
```

This must be generated by the initial analysis AI call.

### `organization.websiteUrl`

The primary submitted or canonical source URL.

This powers:

```text
{{organizationUrl}}
```

### `voiceTone.previewSummary`

A concise 1–3 sentence voice/tone summary for Preview mode.

This powers:

```text
{{voiceTonePreviewSummary}}
```

This must be generated by the initial analysis AI call.

Do not make a separate AI call from the workbench to summarize voice/tone guidance.

### `voiceTone.fullGuidance`

The longer scan-derived voice/tone guidance used in Full Guidance mode and exports.

This powers:

```text
{{voiceToneFullGuidance}}
```

This must be generated by the initial analysis AI call.

## Required option token context

Create a single token context object before assembling preview or full guidance.

Use a helper/service function with a name similar to:

```js
buildGuidanceTokenContext(run, guidanceData, selections, workbenchConfig)
```

Adjust parameter names to match the existing codebase.

The token context should include both analysis tokens and option tokens.

Minimum token context:

```js
{
  // Analysis tokens
  organizationName,
  organizationShortName,
  organizationUrl,
  detectedOrganizationType,
  voiceTonePreviewSummary,
  voiceToneFullGuidance,

  // Option tokens
  guidanceTask,
  guidanceTaskLabel,
  lengthDetail,
  lengthDetailLabel,
  readingLevel,
  readingLevelLabel,
  readingLevelDescription,
  bestPracticePack,
  bestPracticePackLabel
}
```

## Token source rules

### Analysis token rules

Analysis tokens must come from the stored analysis/guidance artifact.

Do not derive `organizationShortName` in the workbench.

Do not derive `detectedOrganizationType` in the workbench.

Do not call AI from the workbench to generate `voiceTonePreviewSummary`.

If required analysis tokens are missing, treat the run as invalid/incomplete and show a clear development/admin-facing error.

Because there is no backward compatibility requirement during active development, do not add fallback chains for abandoned artifact shapes.

### Option token rules

Option tokens must come from the current workbench selections and option configuration.

For example:

```js
guidanceTask: "rewrite_existing",
guidanceTaskLabel: "Rewriting existing text",

lengthDetail: "about_same",
lengthDetailLabel: "About the same length",

readingLevel: "general_adult",
readingLevelLabel: "General Adult",
readingLevelDescription: "grade 6–8",

bestPracticePack: "fundraising_appeals",
bestPracticePackLabel: "Fundraising / donor appeals"
```

Default Guidance Block text should generally use readable label tokens, not raw internal values.

{{voiceToneFullGuidance}}

Do not add lots of speculative tokens yet.

If another token is needed for the current default Guidance Blocks, add it deliberately and document it.

## Token replacement function

Create a small token replacement helper with a name similar to:

```js
renderGuidanceTemplate(templateText, tokenContext)
```

It should:

1. Replace supported `{{tokenName}}` values.
2. Make unknown tokens visible during development/QA.
3. Avoid silent failures.
4. Be covered by basic tests if the project has test conventions in place.

Preferred behavior for unknown tokens:

```text
[[missing:tokenName]]
```

Missing required analysis tokens should also be surfaced clearly, not silently replaced with generic language.

## No generic organization wording

Do not write default templates that say:

```text
this organization
```

when an organization token should be used.

Prefer:

```text
{{organizationName}}
```

or:

```text
{{organizationShortName}}
```

Examples:

```text
You are rewriting existing text for {{organizationName}}.
```

```text
Revise the text so it sounds more like {{organizationShortName}}'s public communications.
```

## No AI calls during token replacement

Token replacement must be deterministic.

Do not call the AI API during token replacement.

Changing workbench options should update tokens and assembled output immediately without an AI call.

## Preview and Full Guidance

The token system should work for both:

```js
assembleGuidancePreview(...)
assembleFullGuidance(...)
```

Both assembly paths should build a token context and render any Guidance Block/template text through the same token replacement helper.

Preview should generally use:

```text
{{voiceTonePreviewSummary}}
```

Full Guidance should generally use:

```text
{{voiceToneFullGuidance}}
```

## Security / escaping

Be careful about rendering token-replaced content into HTML.

If token-replaced guidance is displayed in the browser, it must be escaped or rendered safely according to the existing frontend conventions.

Do not use unsafe HTML injection for token-replaced text unless the content has been explicitly sanitized and the project already has a safe rendering pattern.

Plain text and Markdown downloads should not include HTML markup.
